# Product Roadmap

Date: 2026-06-13

This roadmap translates the evergreen product strategy into a dated execution plan for the current prototype phase.

## Current Position

The CutList already has the core loop in place:

- the LLM proposes,
- metadata providers verify,
- deterministic constraints enforce,
- and the desktop app stores the local draft.

The next stage is not broadening scope. It is making the curation workbench more reliable, more inspectable, and more useful on hard playlist-editing tasks.

## Roadmap Goals

### 1. Improve evidence quality

The current product vision depends on distinguishing verified fact from curator judgment. The current roadmap assumes a product split between Verified rules and Curator guidance. That means richer evidence should come before broader AI behavior.

Focus areas:

- strengthen metadata support for BPM, vocalist profile, energy, and transition evidence,
- make unknown-evidence states more explicit,
- improve confidence reporting when provider data is incomplete or ambiguous,
- expand fixtures for ambiguous matches, wrong versions, and weak metadata.

Current note:

- BPM enrichment is intentionally deferred for now. AcousticBrainz is available but too slow and too stale for the desired user experience, and SongBPM does not permit automated access under its terms. Until a compliant, responsive source is chosen, BPM remains a verified rule only when data is already known.

Success looks like:

- more constraints can be enforced or explained with concrete evidence,
- review suggestions can say what is verified versus inferred,
- rejected candidates are easier to understand and resolve.

### 2. Harden intent modeling

The hybrid intent layer is already promising: deterministic extraction handles explicit constraints while the LLM parses broader curator instructions. Hardened intent modeling v1 is now in place with layered intent output for `operationIntent`, `verifiedRules`, `curatorGuidance`, and `scopeIntent`, plus first-class `replace` routing. The next step is making those semantics more accurate on mixed or ambiguous requests and expanding fixture coverage around contradictions.

Focus areas:

- additive versus persistent constraint interpretation,
- removal versus replacement intent,
- requests that combine shaping and generation,
- better handling for target track counts and total-length requests,
- stronger tests for conversational edge cases.

Success looks like:

- fewer surprising persistent rules,
- fewer requests routed to the wrong workflow,
- better preservation of the user’s actual editing goal.

### 3. Expand transformation review

Transformation Review v1 is in place. The next phase should make it more actionable and more central to the product.

Focus areas:

- clearer role and transition explanations,
- stronger repair suggestions for weak bridges, endings, and pacing cliffs,
- better confidence and risk framing for subjective judgments,
- easier verify-and-apply flows for candidate suggestions,
- richer example playlists and evaluation fixtures.

Success looks like:

- users can understand what the playlist is trying to be,
- repair suggestions feel specific rather than generic,
- review becomes a reliable editing surface, not just commentary.

### 4. Add playlist compression

Compression is one of the clearest differentiators in the product strategy and fits the workbench model well.

Focus areas:

- reduce long playlists while preserving identity, arc, and defining tracks,
- explain what was preserved and what was removed,
- allow safe manual review before application,
- ensure replacements or bridges still pass verification.

Success looks like:

- a user can turn an overlong draft into a sharper one without losing its character,
- compression feels like editing, not replacement.

### 5. Expand discovery-radius behavior

The first discovery-radius pass is now in place: playlists store a default mode, the composer exposes the control, request text can override it one-shot, and candidate prompts use the effective radius. The next phase should validate whether the modes produce meaningfully different but still reliable candidate sets.

Focus areas:

- compare safe versus experimental prompt behavior on real playlist fixtures,
- decide whether review follow-ups should honor one-shot overrides more broadly,
- improve language around what identity the model should preserve as it explores,
- test how far exploration can go before verification and fit quality degrade.

Success looks like:

- users can deliberately choose how conservative or exploratory the curator should be,
- adventurous requests still stay within the verified-workbench model.

## Milestone Order

### Near term

- improve evidence support and unknown-evidence handling,
- harden intent parsing and workflow routing,
- expand evaluation fixtures for real failure cases,
- improve rejection, confidence, and provenance language in the UI.

### Medium term

- deepen transformation review,
- ship playlist compression,
- add discovery radius controls,
- produce lightweight evaluation reports for representative curated scenarios.

### Later

- explore durable taste models derived from accepted, rejected, and reordered tracks,
- consider richer evidence sources only if they improve reliability or explainability,
- revisit accounts, databases, or streaming-service integrations only after an explicit architecture change.

## Non-Goals For This Roadmap

- no account system,
- no OAuth,
- no direct playlist upload,
- no database-first redesign,
- no vector database unless retrieval becomes clearly necessary,
- no expansion into a general recommendation app at the expense of playlist curation.

## Immediate Next Steps

1. Audit which current constraints depend on metadata fields that are usually null.
2. Build a dated evaluation set from representative playlist requests and failure cases.
3. Prioritize the highest-friction intent ambiguities and add targeted tests.
4. Define the first compression workflow as a non-mutating review-and-apply flow.
